## 2. How the bump was done (pin + overlay — reusable checklist)

We do **not** merge Hermes; we clone a tag, apply `patches/`, overlay the engine. The bump is:

```bash
# 1. clone the new tag fresh and confirm our patches still apply
git clone --depth 1 --branch v2026.6.19 https://github.com/NousResearch/hermes-agent.git /tmp/hermes-v017
for p in patches/[0-9]*.patch; do
  git -C /tmp/hermes-v017 apply --check "$p" && echo "ok $p" || echo "REBASE $p"   # --reverse --check => already-present
done
# 2. resolve any REBASE: drop if upstreamed/declined, else git diff -- <file> against the new tree
# 3. bump the pin everywhere:
#    engine-manifest.yaml (pinned_version/pinned_tag/pinned_sha), patches/apply.sh PIN,
#    patches/README.md, scripts/build-engine-image.sh fallback, INSTALL.md, docs/supply-chain.md,
#    docs/okf/guide-{1,2}, scripts/framework_init.py fallback, templates/pack/{PLACEHOLDERS.md,new-pack.sh}
# 4. make check
# 5. bash scripts/build-engine-image.sh    # clones the pin, runs apply.sh, builds; verifies pinned_sha
# 6. smoke-test the image (§5)
```

`patches/apply.sh` globs `patches/[0-9]*.patch`, so dropping/adding a patch file needs no list edit. The pin SHA is verified against the clone by `build-engine-image.sh`.

### Why 07 was dropped (upstreamed)
v0.17.0 implements 07's exact behavior natively. `cron/scheduler.py`'s `_scan_assembled_cron_prompt` now selects a **looser** pattern tier (`_scan_cron_skill_assembled`) when the assembled prompt contains runtime-injected DATA (`has_injected_data=True` — set when a prerun-script digest or an upstream-job output is injected): command-shape patterns are dropped, only unambiguous injection directives block, and the raw user prompt is still strict-scanned separately. That is patch 07's purpose verbatim ("a digest that *describes* `cat ~/.aws/credentials` must not trip the strict scanner"), implemented more completely. Carrying our version would only fight upstream.

### Why 08 was dropped (declined — stay on upstream's model)
Patch 08 avoided the image's recursive `chmod -R`/`chown -R` finalization (which can copy-up-stall on overlay2-on-ZFS) by building trees as `hermes` + a top-level chmod. v0.17.0 **reworked this area** into a stronger model — root-owned, world-readable, **non-writable (`chmod -R a-w`)** immutable install (the runtime no longer needs to write build trees: `PYTHONDONTWRITEBYTECODE` + prebuilt TUI). Preserving 08 would *forfeit that immutability* (the `a-w` guarantee requires a recursive walk). 08 is **generic infrastructure** upstream actively owns, so carrying a divergence there compounds every release for a build-perf concern that's fixable on our side. **Decision: take upstream's model.** Measured cost: a full image build is **4m58s** on this overlay2-on-ZFS host with the recursive chmod present — no stall. If a future host shows real slowdown, fix it in `build-engine-image.sh` (a BuildKit cache mount / non-ZFS scratch context), never a Hermes patch.

> **Rule of thumb for carried patches:** diverge only for **domain-specific** facts upstream will never carry (e.g. OKF compiling security intel that quotes attacker commands). Never diverge on **generic infra** (image perms, build layout) upstream owns — measure and mitigate on our side instead.

## 4. Upstream behavior changes that touch our deployment

### 4.1 s6 uid-remap rewrite — benign for `$(id -u)`
v0.17.0 stops re-chowning the build trees (`.venv`, `node_modules`, `ui-tui`) to the runtime UID on remap; only `$HERMES_HOME` (`/opt/data`) is remapped. The install tree is left root-owned + world-readable, and the runtime never writes it. Verified live (gateway as `HERMES_UID=$(id -u)=1003`): `/opt/data` → `1003:1003`, imports succeed from the root-owned tree, `/opt/data` writable, 0 startup errors. **Note: the runtime still must run as the configured uid** — `/opt/data` + vault writes depend on it; only the build-tree chown relaxed.

### 4.2 cron-script subprocess env sanitization — verified no-impact
v0.17.0 strips provider credentials + Hermes-managed secrets from the env handed to `--script` cron subprocesses (`_sanitize_subprocess_env`, blocklist in `tools/environments/local.py`): provider API keys/base-urls, `TELEGRAM_BOT_TOKEN`, `TAVILY/EXA/SERPER_API_KEY`, `OPENAI_*`, `LLM_MODEL`. **Audited every `scripts/cron/*.py`:** they read only tuning params (`*_BATCH_SIZE`, `DECAY_*`, `HOT_DAYS`, …), Hermes core vars (`HERMES_HOME`/`HERMES_DATA`), and custom vars (`WIKI_PATH`, `OKENGINE_*`) — **none on the blocklist**, so all pass through.

> **Rule for future cron scripts:** a `--script` cron must not rely on a provider key, `TELEGRAM_BOT_TOKEN`, or a web-search key in its own env — those are stripped. Use a **custom** env var name (not registered in Hermes' provider/optional tables) for any script secret.

### 4.3 cron-plus compatibility — no change
cron-plus binds two `cron.scheduler` symbols, both unchanged in v0.17.0:
- `run_job(job)` → 4-tuple `(success, output_doc, response, error)` — cron-plus `runner.py` unpacks 4. ✅
- `_deliver_result(job, content, adapters=None, loop=None)` — signature matches the call site. ✅

Verified in the built image: `run_job(job)` and `_deliver_result(job, content, adapters, loop)`.
